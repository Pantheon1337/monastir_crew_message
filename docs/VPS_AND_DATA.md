# Данные приложения и развёртывание на VPS

## Где хранятся данные

| Что | Путь (от корня `server/`) | Примечание |
|-----|---------------------------|------------|
| База SQLite | `data/app.db` | Пользователи, чаты, сообщения, заявки в друзья, метки «прочитано» и т.д. |
| Загруженные аватары | `uploads/avatars/` | Файлы изображений; в БД хранится относительный путь |

После перезапуска процесса Node и пересборки клиента **данные не исчезают**, пока не удалены эти файлы и папки. Репозиторий можно переклонировать заново — на проде нужно **копировать или монтировать** `app.db` и `uploads/`, иначе будет «чистая» база.

## Сообщения и шифрование

- **Сообщения сохраняются** в SQLite (таблица сообщений direct-чата) и отдаются по REST при открытии чата.
- **Содержимое сообщений не шифруется на уровне приложения**: в БД лежит обычный текст (как вы его отправили). Защита — **HTTPS** между клиентом и сервером и контроль доступа к файлу БД на сервере.
- **Пароли** не хранятся в открытом виде: используется **хеш** (см. `server/password.js`).
- End-to-end шифрование (как в Signal) **не реализовано**.

## Резервное копирование

Регулярно копируйте:

1. `data/app.db` (или снимок через `sqlite3 .backup`).
2. Каталог `uploads/` целиком.

Оба должны быть **согласованы по времени** (бэкап в одну метку), иначе в БД могут остаться ссылки на отсутствующие файлы аватаров.

## VPS: ориентир на ~10 пользователей

Подойдёт минимальный VPS (например **1 vCPU, 1–2 ГБ RAM**, **20–40 ГБ** диска SSD), ОС Ubuntu LTS.

- **Процесс**: один экземпляр Node (Express + WebSocket в том же процессе).
- **Обратный прокси**: nginx или Caddy — TLS, проксирование HTTP и **Upgrade** для WebSocket на тот же upstream.
- **Переменные окружения**: порт приложения, секреты сессии/подписи, при необходимости путь к `data/` и `uploads/`.
- **Система**: `systemd` unit для автозапуска и перезапуска при сбое.
- **Файрвол**: открыты 22 (ssh), 80/443; приложение слушает только `127.0.0.1` за nginx.

SQLite при такой нагрузке и одном процессе записи обычно **достаточен**.

## Масштабирование к 50–100 пользователям на одной VPS

Остаётся один сервер, но:

| Область | Рекомендация |
|---------|----------------|
| Ресурсы | **2 vCPU, 2–4 ГБ RAM**, диск по росту `app.db` и `uploads` |
| SQLite | Один процесс Node — по-прежнему нормально для десятков активных пользователей при умеренном трафике. Следите за **размером БД** и временем бэкапов. |
| WebSocket | Убедитесь, что прокси держит **достаточные таймауты** и буферы для WS; один процесс держит все соединения в памяти — при росте смотрите на лимиты `ulimit`/память. |
| Статика | Раздавайте собранный фронт и `/uploads` через nginx с кэшем для статики; API — на Node. |
| Логи | Ротация логов, чтобы не забить диск. |

Когда **одного процесса** мало (несколько инстансов за балансировщиком) или нужна **высокая конкурентная запись**, следующий шаг — **PostgreSQL** (или другая СУБД) вместо SQLite и **sticky sessions** или **общий pub/sub** для WebSocket между инстансами. Для 50–100 пользователей на одной машине часто ещё не требуется, если нет экстремальной нагрузки на чаты.
